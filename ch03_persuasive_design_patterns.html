導論：數位行為改變支援系統

Introduction: Digital Behavior Change Support Systems

數位健康介入正在深刻地改變醫療照護的面貌。隨著智慧型裝置與網路的普及，Behavior Change Support Systems (BCSSs)——行為改變支援系統——成為健康促進與說服性技術研究的核心議題。

BCSSs 核心定義 Oinas-Kukkonen, 2013

行為改變支援系統是一種社會—技術資訊系統，具有心理與行為層面的預期結果，旨在形成、改變或強化使用者的態度、行為或遵從行為，且不涉及強制或欺騙。

"a socio-technical information system with psychological and behavioral outcomes designed to form, alter or reinforce attitudes, behaviors or an act of compliance without using coercion or deception." — Oinas-Kukkonen (2013)

BCSSs 的興起源於電腦在說服方面相較於人類具有的獨特優勢。Fogg（2002）指出，電腦在說服上的優勢來自以下六項特性：

💪
Persistence
持久性
系統不會疲倦，能持續不斷地傳遞說服訊息
🎭
Anonymity
匿名性
使用者可在匿名情境下揭露敏感行為資訊
📊
Data Management
資料管理
能處理及分析人類無法比擬的龐大資料量
🎨
Modality
多模態性
整合文字、圖像、聲音、影片等多種呈現形式
📈
Scalability
可擴展性
可同時觸及數以百萬計的使用者
🌐
Ubiquity
無所不在性
隨時隨地，透過行動裝置提供即時支援

理論背景：說服性系統設計模型中的對話支援

Theoretical Background: Dialogue Support in the PSD Model

PSD Model (Persuasive Systems Design Model)——說服性系統設計模型（Oinas-Kukkonen & Harjumaa, 2009）——是目前最廣泛使用的 BCSSs 設計理論框架。模型將設計原則分為四大類別：

🎯

主要任務支援

Primary Task Support

支援使用者完成其主要任務或目標的功能設計

💬

對話支援

Dialogue Support

改善使用者與系統之間的溝通，維持使用者的參與動機

🏛️

系統可信度支援

System Credibility Support

讓系統顯得值得信任且具有可信度

👥

社會支援

Social Support

運用社會影響力機制促進目標行為

本章聚焦於 PSD 模型中的dialogue support（對話支援）類別，此類別共包含七項功能，旨在維持使用者對系統的持續投入：

說服的三重脈絡 Three Contexts of Persuasion

PSD 模型強調，有效的說服性系統設計必須考量三種相互影響的脈絡：

🔍

使用脈絡

Use Context

問題領域的性質（如健康、環保、教育），決定系統應達成哪種行為改變目標

👤

User Context

⚙️

軟體設計模式：概念基礎

Software Design Patterns: Conceptual Foundation

設計模式定義 Gamma et al., 1994 — "Gang of Four"

設計模式（design patterns）是針對軟體設計中常見問題的可重複使用解決方案。它們不是可直接執行的程式碼，而是解決問題的模板（templates）或最佳實踐的文件化描述。

"reusable solutions to commonly occurring problems in software design; not programming code but templates." — Gamma et al. (1994)

設計模式的四項本質特徵 Four Essential Characteristics of a Pattern

  1. 模式名稱 Pattern Name 提供一個高層次抽象的共通術語，讓開發者可以在更高的抽象層次上進行溝通，不必每次都從零解釋設計思路。
  2. 問題 Problem 描述何時應套用此模式及其適用脈絡。清楚界定問題是有效選用模式的前提——不同的問題可能需要不同的模式。
  3. 解決方案 Solution 對元素（類別、物件）排列方式的抽象描述，說明如何組裝這些元素來解決問題。解決方案是模板，而非特定的程式碼實作。
  4. 後果 Consequences 套用該模式的結果與取捨（tradeoffs）。了解後果有助於開發者判斷模式是否適合其特定情境，並預期可能遇到的限制。

BCSSs 的架構基礎：MVC 與 REST MVC & REST Architecture

MVC 架構 Model-View-Controller

MVC 將軟體系統分為三個相互分離的層次，使程式碼更易於維護與擴展：

Model（模型） · View（視圖） · Controller（控制器）
  • 📦 Model：資料層，管理應用程式的資料與商業邏輯
  • 🖥️ View：視圖層，使用者介面（UI）的呈現
  • ⚙️ Controller：控制層，處理使用者輸入並協調 Model 與 View

REST 架構 Representational State Transfer

REST 以 URL 識別應用程式的各個資源，並透過四種標準操作處理資源：

Create Read Update Delete

CRUD 操作（建立、讀取、更新、刪除）對應 HTTP 的 POST、GET、PUT/PATCH、DELETE 方法。

🔑 任何 BCSS 的基礎資源：
① User 資源——使用者個人資料、行為資料
② Entry 資源——行為紀錄（如體重數據、運動記錄）

💡 為何設計模式對 BCSSs 開發至關重要？透過記錄與分享已驗證的解決方案，設計模式有助於實現「不重複發明輪子」（don't reinvent the wheel）的工程原則，同時在行為科學意圖與軟體工程實作之間架起橋樑。這有助於提升 BCSSs 開發的標準化程度，使不同系統的研究結果更具可比性。

設計模式一：即時回饋

Design Pattern 1: Instant Feedback

即時回饋模式 Instant Feedback Pattern

對話支援 稱讚 建議 PSD Dialogue Support

系統應在使用者提交行為紀錄時，立即給予關於其表現的即時評語（real-time comments on user performance），使使用者能夠即時了解自身行為是否符合目標，並獲得適當的正向強化或修正建議。

User（使用者） + Entry Controller（紀錄控制器）

當使用者在 BCSS 中建立一筆新的 Entry（行為紀錄）時，Entry Controller（紀錄控制器）的 Create action（建立動作）會立即分析該紀錄。依據分析結果，將適當的回饋訊息（稱讚良好表現，或針對不足提出改善建議）傳遞至 View（視圖）呈現給使用者。

後果與影響 Consequences

✅ 正面後果：使用者每次提交紀錄時，都能收到適切的回饋；有助於使用者重新審視自身進展（reexamine progress）並調整行為習慣。即時性強化了行為—結果的連結，提升學習效果。

⚠️ 實作考量：回饋訊息的品質取決於 Entry Controller 中分析邏輯的設計精細程度。過於簡單的判斷規則可能產生不適切甚至誤導性的回饋，反而損害使用者信任。

設計模式二：提醒

Design Pattern 2: Reminder

提醒模式 Reminder Pattern

對話支援 提醒功能 習慣養成 PSD Dialogue Support

系統應持續主動接觸使用者（continuously reach out to users），強化使用者定期造訪系統的習慣，防止中途放棄（attrition），特別是在使用者一段時間未登入後更需要適時介入。

User（使用者） + Mailer（郵件發送器） + Scheduler（排程器）

User 元件保存使用者個人資料與聯絡資訊（電子郵件地址等）；Mailer 元件包含各種提醒傳遞方法，包括：週期性電子報（weekly newsletter）、閒置觸發提醒（triggered reminder after inactivity）、以及基於行為檔案的個人化提醒（personalized reminders based on behavioral profiles）；Scheduler 元件依設定的時間表自動執行提醒任務。

後果與影響 Consequences

Kelders 等人（2012）的研究證實，提醒功能是預測使用者更高遵從度（better adherence）的重要因素。

✅ 正面後果：定期的提醒電子郵件或簡訊使使用者持續投入系統；支援習慣養成，防止使用者流失（dropout）。個人化提醒比通用提醒更具效果。

⚠️ 負面風險：過於頻繁的提醒反而可能使使用者感到厭煩（irritate users），導致退訂或反感。提醒的間隔與頻率必須審慎評估（carefully evaluated），以在維持參與度與避免騷擾之間取得平衡。

"Kelders et al. (2012) found that reminders predict better adherence, but too-frequent reminders can irritate users; interval must be carefully evaluated."
Kelders 等人（2012）發現提醒功能能預測更高的遵從度，但過於頻繁的提醒可能使使用者感到厭煩——提醒間隔必須審慎評估。

設計模式三：獎勵

Design Pattern 3: Reward

獎勵模式 Reward Pattern

對話支援 獎勵機制 遊戲化 PSD Dialogue Support

系統應給予使用者虛擬獎勵（virtual rewards），進一步激勵其持續參與，並協助目標設定（assist goal-setting）。獎勵機制對應遊戲化（gamification）設計原則，透過成就感維持使用者的長期投入。

User（使用者） + Entry（行為紀錄） + EntryObserver（紀錄觀察者） + Reward（獎勵） + Accomplishment（成就）

User 與 Reward 之間透過 Accomplishment 模型（成就模型）建立多對多關聯（many-to-many relationship）——一位使用者可以獲得多個獎勵，一個獎勵也可被多位使用者取得。EntryObserver 採用觀察者設計模式（Observer design pattern），觀察何時有新的 Entry 被提交；一旦觸發，即檢查該筆紀錄是否使使用者達到獲獎條件；若符合，則建立一筆 Accomplishment 紀錄連結使用者與其獲得的獎勵。

後果與影響 Consequences

✅ 正面後果：給予使用者獎勵能激勵持續參與（motivate continued participation）；協助使用者建立和追蹤目標；成就感與里程碑的累積增強自我效能感（self-efficacy）。

⚠️ 個體差異：並非所有使用者都認為虛擬獎勵同等吸引人（not all users find rewards equally desirable）。部分使用者可能對遊戲化元素反感，因此獎勵機制的設計應考量目標族群的個體差異與偏好。

範例實作：「別中斷鏈條」原型系統

Example Implementation: "Don't Break the Chain" Prototype

Jerry Seinfeld 生產力法則 Seinfeld Productivity Method

「別中斷鏈條」（Don't Break the Chain）原型系統以美國喜劇演員 Jerry Seinfeld 的生產力方法為基礎：選擇一個習慣，每天實踐它，並在日曆上標記當天。連續的標記形成一條「鏈條」（chain）。

"Choose a habit, perform it daily, mark each day → build a 'chain.' The longer the chain, the more resilient the user is in maintaining behavior."

鏈條越長，使用者維持該行為的韌性（resilience）便越強——中斷鏈條的心理代價足以作為強大的激勵因素，防止使用者放棄目標行為。這一機制本質上融合了習慣養成（衝動歷程）與目標追蹤（反思歷程）兩種機制。

技術實作細節 Technical Implementation

技術框架 Technology Stack

原型系統以 Ruby on Rails 開發——這是一個原生支援 MVC 架構與 REST 設計原則的 Web 應用程式框架，非常適合展示本章提出的三種軟體設計模式。

Ruby on RailsMVCREST開源框架

三種模式的整合應用 Integration of Three Patterns

  • ✅ 即時回饋：使用者每次標記完成當日習慣後，系統立即顯示稱讚（表現良好）或鼓勵訊息（需加油）
  • ✅ 提醒：系統透過排程器在使用者閒置時自動發送電子郵件提醒，鼓勵回訪並繼續維持鏈條
  • ✅ 獎勵：達到特定鏈條長度里程碑（如連續 7 天、30 天）時，EntryObserver 觸發成就獎勵

📌 原型的研究意義：「別中斷鏈條」原型展示了三種設計模式在真實 BCSS 脈絡中的可應用性（applicability in a prototype BCSS）。然而，原型層級的驗證只是第一步——模式仍需在完整功能系統中進行更嚴謹的測試，才能確立其實際效用。

討論與結論

Discussion & Conclusions

研究貢獻 Contributions

目前 BCSSs 的開發缺乏清晰、定義明確且具彈性的設計模式（clear, well-defined, flexible design patterns）。本章提出的三種設計模式——即時回饋、提醒、獎勵——試圖填補這一空缺，在行為科學的說服理論與軟體工程的實作實踐之間架起橋樑。

知識分享與最佳實踐文件化 Knowledge Sharing & Documentation

設計模式對於知識分享（knowledge sharing）與記錄最佳實踐（documenting best practices）至關重要。透過模式化描述，開發者無需從零開始，可直接套用已驗證的解決方案，顯著降低開發成本與風險。

標準化與可比性 Standardization & Comparability

設計模式有助於 BCSSs 開發的標準化，使不同系統間的比較成為可能。解決「黑箱問題」，讓研究者能清楚了解系統內部機制，進而提升研究結果的可類化性（generalizability）。

研究限制 Limitations

⚠️ 限制一：三種模式目前僅在原型層級（prototype level）進行概念化，尚未在完整功能系統（fully functional systems）中接受嚴謹測試。

⚠️ 限制二：本章僅聚焦於 PSD 模型中對話支援類別的三項功能，其他類別（如社會影響力模式 social influence patterns）的設計模式仍有待未來研究補充。

未來研究方向 Future Directions

  1. 原生行動應用程式測試 Testing in Native Mobile Apps 模式應在原生行動應用程式（native mobile apps）中進行測試，以驗證其在行動裝置脈絡下的適用性。
  2. 感測器系統整合 Integration with Sensor Systems 與行動裝置或穿戴式裝置的感測器系統連結，實現更精確的行為偵測與即時回饋，強化即時回饋與提醒模式的效能。
  3. 彈性評估 Evaluating Flexibility 評估模式在不同 BCSSs 脈絡中的彈性，確認設計模式是否能有效跨越不同健康行為領域（如運動、飲食、戒菸）進行應用。
  4. 開發者溝通改善評估 Evaluating Communication Improvement 實證評估模式是否確實改善了行為科學家與軟體工程師之間的溝通效率，以及是否降低了 BCSSs 開發的複雜性與成本。
  5. 其他 PSD 類別模式化 Patterns for Other PSD Categories 將模式化方法擴展至 PSD 模型的其他三個類別（主要任務支援、系統可信度支援、社會支援），建立完整的 BCSSs 軟體設計模式庫。

核心結論 Key Conclusions

軟體設計模式為 BCSSs 開發提供了一個具體且可操作的橋樑，連接行為科學的說服意圖與軟體工程的實作執行。透過記錄、分享並標準化這些模式，研究者與開發者可以更有效率地建構更有效的行為改變支援系統——不必每次都重新發明輪子，而是站在前人經驗的肩膀上持續創新。

"Patterns are important for knowledge sharing and documenting best practices … they enhance standardization of BCSS development and bridge behavioral science intent with software engineering implementation." — Oduor, Alahäivälä & Oinas-Kukkonen (2017)
